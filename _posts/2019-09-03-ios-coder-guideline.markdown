热力学第二定律告诉我们在自然过程中，一个孤立系统的总混乱度（即“熵”）不会减小。就像如果我们不进行定期打扫房间，那么不久我们的房间就会变得乱糟糟。在程序员日常编码过程中，如果不对自己编写的代码进行约束和统一，就会导致项目整体的可阅读性和可维护性偏低。因此需要一份规范文档来对自己编写的代码进行约束和统一。

### 项目结构

* Classes（类文件主目录）
    * CoreBLL（核心业务层）
    * Commom（常用组件层）
        * Model（常用模型）
        * Util（工具类）
    * Expand（扩展层）
        * Category（分类）
        * Marocs（头文件）
        * Vendor（第三方库）
    * Managers（核心协调层）
        * Localization（本地化）
        * Mediator（Toast以及弹窗）
    * Network（网络层）
        * Operation（接口）
        * StatusCode（通用状态码）
    * Pages（表现层）
    * UILibrary（UI组件层）
        * Widget（组件）

### 命名规范

好的命名规范能够增加代码可读性和可维护性，在多人合作开发的项目中和人员变动较大的 项目中，统一命名约定是必不可少的。所有的命名必须使用英文，不可出现拼音，数字，单字母，汉字等无意义或无法读懂的命名(有特殊意义的除外，如:A2)。

#### 源码文件命名

源码文件都必须以公司名或项目名大写缩写作为前缀，譬如JP(JollyPay)(前提是全部统一)，然后接其名字，名称遵循大驼峰命名法。

#### 类命名

所有的类名、接口名(Protocol)均以缩写作为前缀，名称遵循大驼峰命名法，不可出现下划线( _ )等特殊字符。类名、接口名必须是有意义的。

> 类名 = 缩写 + 类描述 + 类型

``` objective-c
JPHomeViewController = JP + Home + ViewController;
JPRecordTableViewController = JP + Record + TableViewControler;
```

#### 方法命名

方法名应全部使用有意义的单词组成，不允许使用 get 作为前缀，- 或 + 与返回类型之间留一个空格，方法名遵循小驼峰命名法，不可出现下划线( _ )等特殊字符。

#### 常量命名

常量使用小写 k 作为前缀，名称遵循大驼峰命名法。常用的宏可使用全大写字母命名，例如: CURRENTUSER 。

过长的宏也是用小写k+大驼峰(优先推荐)。

枚举使用项目名大写缩写作为前缀，名称遵循大驼峰命名法(必须遵循 OC 的写法，不允许出现 C 语言枚举的写法)。

常量包括:宏、枚举、常量。

`注:除全大写的宏可包含下划线( _ )外，其他常量不可使用特殊字符。`

#### 属性、实例变量命名

* 尽量使用属性而非实例变量。
* 属性、实例变量命名采用小驼峰命名法(后台返回数据由于不可抗原因，可使用下划线命名法)，必须见名知意，不可使用 OC、XCode 关键字(如id等)，实例变量必须使用下划线( _ )作为前缀。

`注:外部类不需要使用的属性、实例变量，必须声明为私有，外部类需要使用的才声明为公开。`

### 代码复用和容错性

1. 常用的代码必须写成宏。
2. 常用的方法必须封装成工具类或分类。
3. 一个方法不能实现多个功能或多个操作。
4. 有返回值的方法中，每一个退出点都需要有返回值。
5. 有参数的方法中要对输入参数的正确性和有效性进行检查。
6. 对象赋值之前必须判断值是否为空。
7. 方法的参数不宜过多。
8. 为了更好适配，优先使用 OC 数据类型而非 C 语言数据类型(如: NSInteger、CGFloat 等)。
9. 测试数据必须在测试结束后及时删除(测试时可用 // ToDo delete 标记)。

Follow your heart.
